Git

Sistema de control de versiones distribuido que se basa en la idea de que cada desarrollador tiene su propia copia completa del repositorio, lo que permite un desarrollo descentralizado, un historial de cambios eficiente y la capacidad de fusionar cambios de diferentes ramas.


Resumen: Control de versiones, mostrar historial, rastrear/visualizar, alterar código.

  • Repositorio. Carpeta que deja de existir en local, que lo subo a un servidor para que todas las carpetas que sepan de su existencia, puedan hacer copias, son copias que viven directamente sobre el repositorio.

Lista de comandos

  • pwd. print work directory. Nos ayuda a imprimir nuestro directorio completo activo.
  • ls. list directory contents. Nos muestra que hay dentro de lo que incluye la carpeta en la que estamos posicionados.
  • ls -l lista toda la información que encuentre, con un formato más largo.
  • ls -alR R significa recursivo. Podemos ver detalladamente los archivos de la carpeta en la que estamos posicionados.
  • history Revisar el historial de comandos utilizados.
  • clear Limpiar la información de la ventana.

Estructura general de un comando

$ qué quiero hacer / cómo lo voy hacer / a quién lo voy hacer

Comandos básicos para manejar directorios

  • cd (change directory). Para cambiar entre directorio y para salir de todo.
  • cd + letra inicial de la carpeta + tab autocompleta.
  • cd .. Regresa a un nivel anterior.
  • cd nombreDelDirectorio para entrar a un directorio en un nivel siguiente. (cd /nombredeldirectorio/otrodirectorio/otrodirectorio)
  • mkdir (make directory): Para crear directorios.
  • rmdir (remove directory): Para eliminar directorios VACÍOS.
  • cp (copy file): cp ArchivoACopiar.ext../ Para copiar archivos.
  • mv (move file): Para mover archivos.
  • rm (remove file): Para remover archivos.
  • rm -r nombreDelArchivo Borra todo lo que encuentre (tanto la carpeta como los archivos dentro)
  • touch + nombre con extensión: Creación de archivos

Para crear un documento en nano

  1. Ejecutar nano ejemplo.txt
  2. Escribe la información dentro del archivo.
  3. Cuando estemos listos de guardar, mantén presionada la tecla Ctrl y presiona la letra O. Nos preguntará si queremos guardar el archivo.
  4. Damos enter y se guarda.
  5. Cuando terminemos, presionamos Ctrl + X para cerrar nano, y regresar a la terminal.

GitHub

Plataforma para poder crear proyectos abiertos y que se va caracterizar principalmente por las funciones colaborativas.

¿Cuál es la diferencia entre Git y GitHub?

Git es el sistema de control de versiones que permite realizar un seguimiento de los cambios en el código, mientras que GitHub es una plataforma en línea que utiliza Git para alojar, colaborar y administrar proyecos de desarrollo de software de manera colaborativa.

  • git --version para saber qué versión de git tenemos intalada.
  • git help para obtener ayuda en comandos git.
  • git config --global user.name "NombreDeUsuario"
  • git config --global user.email correoelectronico.com
  • git config --list para validar los datos ingresados.
  • git init para convertir mi carpeta local a repositorio.
  • git status para revisar el estado de mi repositorio.
  • git add . para agregar todos los archivos.
  • git commit para agregar comentarios/para trabajar en vim, i=para agregar información, damos esc para salir y presionamos :wq para guardar y salir.
  • git push subir los cambios al repositorio de GitHub.
  • git remote add origin direccionDelRepositorio.git
  • git commit -m "mensaje" se agrega el comentario entre las comillas.
  • git log para saber el historial de comentarios.

JavaScript

Ciclos o bucles (loops)

También conocidos como bucles, son estructuras de control que nos van a permitir ejecutar un bloque de código de forma repetida hasta que se cumpla una condición específica.

Es importante usar ciclos para automatizar tareas repetitivas. En JavaScript existen los siguientes ciclos:

  • While
  • Do While
  • For

Iterar. Es el proceso de repetir una secuencia de instrucciones.

Estructura de control. Herramienta especial para crear algoritmos o programas más dinámicos y flexibles, nos ayuda a controlar la ejecución del código.

While

Ejecuta un bloque de codigo mientras se cumple una condición específica. Es una estructura de control que se utiliza para repetir un bloque de código mientras una condición sea verdadera. 

                while (condición) {
                    // Código que se ejecuta mientras la condición sea verdadera
                }

Donde:

  1. Condición. Es una expresión booleana que se evalúa antes de cada iteración del bucle. Si la condición es verdadera, el bloque de código dentro del bucle se ejecutará; de lo contrario, el bucle se detendrá y la ejecución continuará con el código después del bucle "while".
  2. Bloque de código. Es el conjunto de instrucciones que se ejecuta mientras la condición sea verdadera.
Ejemplo. Consultorio dental 
                let capacidadPacientes = 10;
                let pacientesEnConsultorio = 1;

                //mientras 1 sea menor que 10...
                while (pacientesEnConsultorio < capacidadPacientes){
                    // seguiré dando servicio
                    console.log("Seguimos dando servicio");
                    // y aumento en uno en uno a mis pacientes
                    pacientesEnConsultorio++;
                }

Do While

Se ejecuta la instrucción, mientras sea verdadero. (Primero ejecuta, después pregunta y si es verdadero, vuelve a ejecutar).

El bucle do while se diferencia del while ya que aquí se ejecuta la instrucción al menos una vez, y luego se pregunta para saber si se sigue ejecutando o no. 

                do {
                    // código a ejecutar
                } while (condición);
Ejemplo. Consultorio dental 
                var contadorPacientes = 0;

                do {
                    var nombre = prompt("Ingrese el nombre del paciente");
                    if (nombre && (contadorPacientes < 15)){
                        console.log("Se ha registrado con éxito a " + nombre);
                        contadorPacientes++
                    } else {
                        console.log("Límite excedido de pacientes por día");
                        break;
                    }
                } while (confirm("¿Desea registrar otro paciente?"));

                console.log("Número total de pacientes: ", contadorPacientes);

For

Se utiliza cuando conocemos el número exacto de iteraciones que vamos a realizar. Consiste en 3 partes fundamentales:

  • Inicialización. Se ejecuta antes de que el bucle comience y se usa para establecer una variable de control (variable de inicio).
  • Condición. Es el detonante que verificamos antes de cada iteración, para saber si se sigue ejecutando o no (siempre y cuando sea verdadera).
  • Expresión de iteración (contador). Tiene la finalidad de modificar la variable de inicio y poder tener un punto de cierre. 
                for (inicialización; condición; expresión de iteración) {
                    // bloque de código a ejecutar
                }
Ejemplo. Crea un código que imprima los números del 0 al 4 
                for (var valorInicial = 0; valorInicial < 5; valorInicial++){
                    console.log(valorInicial);
                }
Versión más "comercial" 
                for(var i = 0; i < 5; i++) {
                    console.log(i);
                }
Ejemplo. Consultorio Dental: registro de citas 
                let cantidadDeCitasDiarias = 5;

                for (let i = 1; i <= cantidadDeCitasDiarias; i++){
                    var nombre = prompt("Ingrese el nombre del paciente " + i);
                    console.log("Paciente " + nombre + " con cita " + i);
                }

Scrum

Es un marco de trabajo simple que promueve la colaboración en los equipos para lograr desarrollar productos complejos para satisfacer las necesidades de los clientes.

Metodología tradicional.

Planificación proactiva donde factores como el costo, alcance y el tiempo son importantes. Buscan imponer disciplina.

Metodología waterfall.

Requisitos (qué se necesita para el resultado final), diseño desarrollo, pruebas y lanzamiento. Desventajas: es imposible predecir cada detalle del producto, los dueños no tienen visibilidad del producto hasta su lanzamiento.

Los 4 principios Agile.

  • Individuos e interacciones sobre procesos y herramientas.
  • Software funcionando sobre documentación extensiva.
  • Colaboración con el cliente sobre negociación contractual.
  • Responder al cambio sobre seguir un plan.

Desarrollo de Software Agile.

  • Enfoque a la mejora continua.
  • Software funcional.
  • Adaptabilidad en cambios.
  • Colaboración y transparencia con el cliente para que este pueda conocer su producto y dar retroalimentación del mismo.
  • Enfoque a los individuos e interacciones.

Roles de Scrum.

Product Owner

  • Representa los requerimentos y necesidades del cliente.
  • Es el encargado de definir la visión y dirección del producto.
  • Es responsable de refinar el product backlog.
  • Busca maximizar el valor del producto, para el cliente, los usuarios y la organización.

Scrum Master

  • Se encarga de facilitar la ejecución del marco de Scrum.
  • Es un líder sirviente (es decir, que no da órdenes ni reparte tareas si no apoya y si sirve al equipo para que lleguen a su meta.)
  • Ayuda a eliminar impedimentos que puedan afectar la entrega del producto.
  • Facilita las ceremonias Scrum.
  • Analiza las métricas del burndown chart para buscar procesos de mejora.

Equipo de Desarrollo

El equio de desarrollo es un equipo multifuncional. Es decir, lo integran desarrolladores, testers y tienen todas las habilidades técnicas necesarias para desarrollar el proyecto.
Son colaborativos, es decir, ellos mismos organizan sus tareas.

Stakeholders

Son individuos, grupos o entidades que tienen un interés o están involucrados de alguna manera en el proyecto o en el resultado final del producto. Pueden tener influencia sobre el proyecto, ser afectados por él o tener algún tipo de relación con el mismo.
Pueden incluir:

  • Clientes y usuarios finales,
  • patrocinadores,
  • proveedores,
  • reguladores.

Artefactos.

  • Product Backlog. Lista de necesidades, resultado del refinamiento.
  • Sprint Backlog. Plan de acción, resultado del Sprint Planning (primera reunión donde juntamos nuestras ideas.)
  • Increment. Suma al producto, resultado del Sprint.

Historias de usuario.

Título de la Historia

  • Cómo "role, persona". Persona o rol de usuario que tiene la necesidad.
  • Quiero "objetivo, comportamiento". Lo que se requiere obtener: una funcionalidad, característica, etc.
  • Para "motivo, razón, valor". Motivo por el que se necesita, valor que se obtiene como resultado, etc.

Rituales.

  1. Planificación del Sprint (Sprint Planning)
    • El equipo Scrum, que incluye a los desarrolladores, el Scrum Master y el Product Owner, se reúne para planificar el trabajo que se realizará durante el próximo Sprint.
    • El objetivo es definir qué elementos del Backlog del Producto (conjunto de requisitos y funcionalidad pendientes) se incluirán en el Sprint y cómo se implementarán.
    • Al final de esta fase, el equipo debe tener una comprensión clara de lo que espera lograr en el Sprint y cómo medir el éxito.
  2. Desarrollo del Sprint (Sprint Development)
    • Durante el Sprint, el equipo trabaja en la implementación de las funcionalidades seleccionadas.
    • Las tareas se realizan en ciclos cortos y regulares llamados "Daily Scrums" o "Daily Standups", en los cuales los miembros del equipo comparten actualizaciones sobre su progreso, desafíos y próximas tareas.
    • El equipo trabaja de manera autónoma para lograr los objetivos del Sprint, manteniendo un enfoque en la entrega de valor al final del Sprint.
  3. Revisión del Sprint (Sprint Review)
    • Al final del Sprint, se lleva a cabo una reunión de revisión para mostrar al Product Owner y a los interesados las funcionalidades completadas durante el Sprint.
    • El equipo demuestra cómo funcionan las nuevas características y recopila comentarios para realizas ajustes antes de la siguiente iteración.
    • Esta reunión ayuda a garantizar que el trabajo realizado sea coherente con las expectativas y permite a los interesados proporcionar retroalimentación temprana.
  4. Retrospectiva del Sprint (Sprint Retrospective)
    • Después de la reunión de revisión, el equipo realiza una retrospectiva para evaluar el proceso y la colaboración durante el Sprint.
    • Se analizan los aspectos positivos y los problemas que surgieron, y se identifican oportunidades de mejora para futuros Sprints.
    • El objetivo es adaptar y ajustar continuamente el proceso para aumentar la eficiencia y la efectividad del equipo.
